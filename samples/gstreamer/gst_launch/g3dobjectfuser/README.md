# Camera + LiDAR Object Fusion with `g3dobjectfuser`

This sample shows how to fuse 2D camera detections with 3D LiDAR detections using the `g3dobjectfuser` element. It reuses the PointPillars assets shared with the `g3dinference` sample.

## How It Works

`g3dobjectfuser` is a single-pad `GstBaseTransform` placed **after** `gvastreammux`. The mux merges the camera stream and the LiDAR stream into one `GstAnalyticsBatchMeta` container; the fuser reads the per-stream metadata out of that container, associates the modalities, runs per-modality tracking, and writes the result back in place.

```
000002.png ! pngdec ! videoconvert ! gvadetect          ! mux.sink_0
000002.bin ! g3dlidarparse ! g3dinference                ! mux.sink_1
gvastreammux output-mode=container sync-mode=first-pts ! g3dobjectfuser ! gvametaconvert ! gvametapublish (JSON)
```

Key points:

- **Detection runs per-camera BEFORE the mux.** Once a LiDAR/radar stream joins `gvastreammux` the mux must run in `output-mode=container` and emits a `multistream/x-analytics-batch` buffer.
- The fuser reads `GstAnalyticsODMtd` from the `video/*` stream and `GstAnalytics3DODMtd` from the `application/x-lidar` stream. The 3D sensor is identified by each stream's caps inside the batch.
- Output is the same container, now carrying per-camera `GstAnalyticsTrackingMtd` and cross-modal `IS_PART_OF` relations. `gvametaconvert` serializes it to JSON.

## Prerequisites

### 1. Verify DL Streamer installation

```bash
gst-inspect-1.0 g3dobjectfuser
```

### 2. Prepare the PointPillars demo assets

The sample reuses the assets prepared by the `g3dinference` sample. Run its prepare script once:

```bash
../g3dinference/g3dinference_prepare.sh
```

This downloads/builds the PointPillars models, the OpenVINO extension, and the `pointpillars_ov_config.json`, and copies the demo frame (`000002.bin`, `000002.png`, `000002.txt`) into `../g3dinference/.pointpillars/data`.

### 3. YOLO model

`g3dobjectfuser.sh` downloads the camera detector automatically via `samples/download_public_models.sh` (default `yolo11n`) into `MODELS_PATH` if it is not already present.

## Run

```bash
export MODELS_PATH=~/models     # where YOLO / PointPillars models live
./g3dobjectfuser.sh
```

The script:
1. builds `calib/kitti_000002.json` from the KITTI `000002.txt` calibration (expands `R0_rect` to 4×4 and `Tr_velo_to_cam` to 4×4, keeps `P2` 3×4),
2. downloads the YOLO model if needed,
3. runs the camera + LiDAR fusion pipeline and writes the fused per-stream metadata to `g3dobjectfuser_output.json`.

### Environment overrides

| Variable | Default | Purpose |
|----------|---------|---------|
| `MODELS_PATH` | `../models` next to the repo | Model root for YOLO and PointPillars |
| `YOLO_NAME` | `yolo11n` | YOLO model to download/use |
| `DEVICE` | `CPU` | OpenVINO device for both detectors |
| `PP_CONFIG` | `../g3dinference/.pointpillars/config/pointpillars_ov_config.json` | PointPillars runtime config |
| `OUTPUT_JSON` | `./g3dobjectfuser_output.json` | Output JSON path |
| `FRAME_INDEX` | `2` | KITTI demo frame index |

## Output

`g3dobjectfuser_output.json` contains one record with the batch's streams:

- camera stream (`stream_index` 0): 2D detections (`objects`) with bounding boxes, labels and tracker `id`s. A detection fused with a 3D object also carries `associated_3d_object_id`: the `id` of the matching `objects_3d` entry on the 3D stream;
- LiDAR stream (`stream_index` 1): 3D detections (`objects_3d`), each with an `id` (the cross-modal join key), `bbox_3d` (x, y, z, l, w, h, yaw, pitch, roll), `confidence`, `label_id`, and `modality: "lidar"`.

## Files

- `g3dobjectfuser.sh` — camera + LiDAR fusion sample (this directory's entry point).
- `calib/kitti_000002.json` — calibration generated by the script from `000002.txt`.

For element details, properties, and output metadata, see [docs/user-guide/elements/g3dobjectfuser.md](../../../../docs/user-guide/elements/g3dobjectfuser.md).
